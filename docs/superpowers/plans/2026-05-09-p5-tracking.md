# P5: Tracking Implementation Plan

> Synthesized from CodeRabbit's per-issue plans on Issues #19–#22, adjusted to fit on top of P4's already-implemented `/ads/serve`, `/ads/click`, `/ads/image`. CodeRabbit's plans assumed those endpoints didn't exist yet.

**Goal:** 公開配信エンドポイントに impression/click イベント記録を追加し、bot UA フィルタ・5 分 dedup・daily salt rotation 基盤を整備。スポンサー向け統計を `ad_stats_daily` VIEW で効率化。

**Architecture:** `src/utils/` に pure ユーティリティ群（salt, dedup, event-filter）を追加。`src/serve/` の既存 P4 ハンドラ（router, click）に `trackEvent` 呼び出しを差し込む。新マイグレーションで `ad_stats_daily` VIEW を作成。`getAggregateStats` は既存だが placeholder 除外は既に対応済み — VIEW 利用に切り替えるか確認。

**Tech Stack:** P4 と同一。新規依存なし。

**スコープ:** P5.1 から呼ばれる pure utility 群。

  - `getDailySalt(client, fallback): Promise<string>` — `system_settings` から `key='ip_hash_salt'` の `value.salt` を読み出す。未設定なら fallback を返す（cron 未実装期は `IP_HASH_SALT_BOOTSTRAP` を fallback）

  - `insertAdEvent` SQL pattern + params

## P5.1 (#19): `ad_events` INSERT from `/ads/serve` and `/ads/click`

**スコープ:** P4 の既存エンドポイントを編集して event 記録を追加。

**変更:**
- `src/serve/router.ts` (編集):
  - 既存の `/serve` ハンドラ末尾、レスポンス返却前に：
    ```
    for each returned regular ad (skip placeholder/house clicks):
      shouldRecord = shouldRecordEvent({ method: 'GET', ua })
      if (!shouldRecord) skip
      ipHash = computeIpHash(ip, dailySalt)
      duplicate = isRecentEvent(client, ad.id, ipHash, 'impression')
      if (duplicate) skip
      ctx.waitUntil(insertAdEvent({ ... event_type: 'impression' }))
    ```
  - HEAD リクエストはそもそも GET ハンドラなので Hono 側で 404。`shouldRecordEvent` は念のためのガード。
  - placeholder は記録しない（spec §5.6 準拠）。house は記録する（admin tool で見れるべき）。
- `src/serve/click.ts` (編集):
  - 302 を返す前に同期的に：
    - bot UA フィルタ（filter）
    - dedup チェック
    - placeholder kind の場合はスキップ（spec §5.6）
    - INSERT INTO ad_events (event_type='click')
  - 失敗してもユーザーには 302 を返す（best-effort、ログは `console.warn`）
- `src/serve/click.ts` の `getAdLinkUrl` を拡張して `kind` も返却（placeholder 判定のため）
- tests:
  - 各種 dedup / filter ケース（実 DB は使わずモック）
  - integration: SELF.fetch で `/click` → 302、event INSERT 試行

**Note:** ad_events への INSERT は P4 と同じ `withPgClient` 経由。元のリクエスト処理を遅延させないよう、impression は `ctx.waitUntil` で fire-and-forget。click は同期（一貫性 > レイテンシ）。

## P5.4 (#22): `ad_stats_daily` VIEW + sponsor stats

**スコープ:** spec §3.6 の集計 VIEW を migration で作成。`getAggregateStats` が既に存在するが（P2.5）、より効率的な VIEW ベースに刷新する選択肢を提供。

**新規 / 変更:**
- `migrations/0004_ad_stats_daily_view.sql`:
  ```sql
  CREATE OR REPLACE VIEW ad_stats_daily AS
    SELECT ad_id,
           date_trunc('day', ts) AS day,
           COUNT(*) FILTER (WHERE event_type = 'impression') AS impressions,
           COUNT(*) FILTER (WHERE event_type = 'click')      AS clicks
      FROM ad_events
     GROUP BY ad_id, date_trunc('day', ts);
  ```
- `migrations/meta/_journal.json` 更新
- `src/db/queries/ads.ts` (編集) — `getAggregateStats` を VIEW 利用に切り替え可能なら切り替え。placeholder 除外（既に `kind <> 'placeholder'` で対応済み）と CTR 計算は維持
- tests: 既存テスト互換性 + 新 VIEW を使ったクエリパターン

**Sponsor の `/ad stats` ボタン UI は既に P2.5 で実装済み**。VIEW 切り替えは内部実装の最適化のみ。
